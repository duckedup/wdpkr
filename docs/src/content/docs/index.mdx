<p class="lede not-content">
  <strong>wdpkr is not a replacement for grep/ripgrep.</strong> It's the
  conceptual layer on top. The agent reads the actual files for ground truth —
  wdpkr's job is to <strong>point and describe</strong>, not to ship source into
  the context window.
</p>

<p class="section-kicker not-content">Why it works</p>

<div class="feature-grid not-content">
  <article class="feature">
    <span class="num">01</span>
    <h3>Embed summaries, not code</h3>
    <p>Off-the-shelf embedders are mediocre on conceptual queries against raw code. LLM-generated summaries close that gap.</p>
  </article>
  <article class="feature">
    <span class="num">02</span>
    <h3>AST-driven chunking</h3>
    <p>Tree-sitter parses files into meaningful symbols — functions, types, traits — across 8 languages, not arbitrary line splits.</p>
  </article>
  <article class="feature">
    <span class="num">03</span>
    <h3>Pluggable backends</h3>
    <p>Traits for VectorStore, Embedder, Summarizer, and Chunker. Swap providers without touching the pipeline.</p>
  </article>
  <article class="feature">
    <span class="num">04</span>
    <h3>CLI, not MCP</h3>
    <p>Any agent that can shell out can use it. JSON to stdout, errors to stderr.</p>
  </article>
</div>
